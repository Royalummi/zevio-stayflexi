# Stayflexi Revised Integration Plan (Post-Call)

> **Date:** After call with Stayflexi  
> **Decision from call:** Stayflexi does **not** host a webhook for property updates.  
> **Zevio must:** Provide **API endpoints + API key (token)**. Stayflexi **calls Zevio** when inventory/rates/restrictions change.  
> **Final XML spec:** `docs/integration/ota_request_response_logs.rtf.md` (+ full OTA doc)

> **“URL: TO be supplied by the consumer”** (Zevio supplies the URL; Stayflexi POSTs XML to it)

We already have:

| What Stayflexi needs | What Zevio already has |
|----------------------|-------------------------|
| API endpoint(s) for XML | `POST /api/channel-manager/stayflexi/webhook` |
| API key / token | `CHANNEL_MANAGER_PROVIDER_STAYFLEXI_SHARED_SECRET` via header `x-channel-secret` |
| Inventory update XML | `UpdateRoomInventoryRQ` — tested ✅ |
| Rates update XML | `UpdateRoomRatesRQ` — tested ✅ |

| Hotel detail | `HotelDetailRQ` — tested ✅ |
| Response `SuccessRS` / `ErrorRS` | Implemented ✅ |

**Automated tests:** 40/40 inbound + scenario tests passing.

## What we deliver to Stayflexi (deliverable pack)

Send Stayflexi a short **Integration API document** with:

### 1) Base URL (zevio.cloud testing)

```
https://api.zevio.cloud
```

### 2) API key (token)

- **Header name:** `x-channel-secret`
- **Value:** generated by Zevio (store in server `.env` as `CHANNEL_MANAGER_PROVIDER_STAYFLEXI_SHARED_SECRET`)
- **Also accepted:** `Authorization: Bearer <token>` (already supported in middleware)

Zevio generates the key. Stayflexi does **not** need to send us a secret.

### 3) API list (operations from their final document)

**Option A — One endpoint (what we have today — recommended)**

Stayflexi sends different XML root elements to **one URL**:

| # | Operation (XML root) | Method | URL |
|---|----------------------|--------|-----|
| 1 | `UpdateRoomInventoryRQ` | POST | `/api/channel-manager/stayflexi/webhook` |
| 2 | `GetRoomInventoryRQ` | POST | same URL |
| 3 | `UpdateRoomRatesRQ` | POST | same URL |
| 4 | `GetRoomRateRQ` | POST | same URL |
| 5 | `UpdateRestrictionRQ` | POST | same URL |
| 6 | `GetRestrictionRQ` | POST | same URL |
| 7 | `HotelDetailRQ` / `GetHotelDetailRQ` | POST | same URL |

- **Content-Type:** `application/xml` or `text/xml`
- **Body:** XML as in `ota_request_response_logs.rtf.md`
- **Success response:** `<SuccessRS />` or operation-specific `*RS` XML
- **Error response:** `<ErrorRS Code="..." />`

**Option B — Separate URL per operation (only if Stayflexi insists)**

We can add **alias routes** that call the same handler (small, safe change). Example:

```
POST /api/channel-manager/stayflexi/inventory/update
POST /api/channel-manager/stayflexi/rates/update
...
```

**Ask Stayflexi:** Is one URL with different XML roots OK, or do they require separate URLs?

### 4) Mapping codes

After pilot property is chosen, Zevio creates mapping in admin:

- Hotel Code ↔ integration `external_hotel_id`
- Room Type Code ↔ mapping `external_room_type_id`
- Rate plan codes ↔ daily controls / mapping metadata

Stayflexi uses **real codes** in XML; Zevio maps to internal `property_id`.

## Open questions — MUST confirm with Stayflexi before changing code

These decide if we need **any** backend changes beyond documentation.

### 🔴 Question 1 — Bookings (most important)

When a guest **books on Zevio**, how does Stayflexi receive the booking?

| Option | Meaning | Our code today |
|--------|---------|----------------|
| **A** | Zevio **POSTs PushBooking XML to Stayflexi’s URL** | Outbound service exists ✅ (Stayflexi OTA doc says “We will expose an endpoint”) |
| **B** | Stayflexi **calls a Zevio API** to fetch new bookings | Not built — would need new APIs |
| **C** | Bookings not synced in phase 1 | Outbound can stay disabled |

**On the call they said Stayflexi has no “webhook”.** That may mean only **inbound property updates** — not necessarily that PushBooking is cancelled.

**Ask Stayflexi exactly:**

> “For booking confirm/cancel: do you give us a PushBooking URL, or do you pull bookings from our API?”

### 🟡 Question 2 — One URL or many URLs?

> “Can you call one endpoint with different XML roots, or do you need a separate URL per operation?”

If one URL is fine → **no code change**.

### 🟡 Question 3 — Do they use GET operations?

Their log file only shows **updates**. Confirm if they will call:

- `GetRoomInventoryRQ`
- `GetRoomRateRQ`
- `GetRestrictionRQ`
- `HotelDetailRQ`

We support them; disable in docs if they don’t need them.

### 🟡 Question 4 — IP allowlist

Do they need their server IPs whitelisted on `api.zevio.cloud`, or is API key enough?

### 🟡 Question 5 — Environment

Confirm `zevio.cloud` for testing, `zevio.in` for production — separate API keys?
